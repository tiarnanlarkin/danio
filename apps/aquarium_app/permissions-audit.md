```dart
// From notification_service.dart
Future<bool> requestPermissions() async {
  final android = _plugin.resolvePlatformSpecificImplementation<
      AndroidFlutterLocalNotificationsPlugin>();
  if (android != null) {
    final granted = await android.requestNotificationsPermission();
    return granted ?? false;
  }
  return true;
}
```

### Notification Permission (POST_NOTIFICATIONS)

**Request Location:** `lib/services/notification_service.dart:40-59`

**Implementation Quality:** ✅ EXCELLENT

**Checks:**
- ✅ Requests permission before enabling notifications
- ✅ Handles user denial gracefully
- ✅ Shows clear error message on denial
- ✅ App remains functional without permission
- ✅ User can retry permission request
- ✅ Test notification feature verifies permission status

**User Experience Flow:**
1. Settings → Enable Notifications toggle
2. System permission dialog appears
3. **If Granted:** Notifications enabled, snackbar confirms
4. **If Denied:** Toggle stays off, snackbar shows "Notification permission denied"
5. User can toggle again to re-request

**Code Review:**
```dart
// From settings_screen.dart:89-107
Future<void> _toggleNotifications(BuildContext context, WidgetRef ref, bool enable) async {
  if (enable) {
    final service = NotificationService();
    await service.initialize();
    final granted = await service.requestPermissions();
    
    if (!granted) {
      if (context.mounted) {
        ScaffoldMessenger.of(context).showSnackBar(
          const SnackBar(content: Text('Notification permission denied')),
        );
      }
      return; // ✅ Doesn't enable notifications if denied
    }
  }
  
  await ref.read(settingsProvider.notifier).setNotificationsEnabled(enable);
  // ✅ Only sets enabled state if permission granted
}
```

**Edge Cases Handled:**
- ✅ Permission denied → toggle stays disabled
- ✅ App restart → permission persists (Android system handles)
- ✅ User can re-request permission later
- ✅ iOS compatibility also implemented
- ✅ Null safety handled

## 🎓 Technical Notes

### Android 13+ Photo Picker Behavior
- **API 33+:** Uses system photo picker by default (no permissions)
- **API 24-32:** Falls back to gallery intent (no permissions still!)
- **Camera access:** User can take new photo from within picker

### Storage Access Framework Benefits
- User has full control over what files app can access
- No ability to scan/list all files
- Matches iOS's privacy-first approach
- Google recommends this approach in Android docs

### Notification Permission Timeline
- **Android 12 and below:** VIBRATE auto-granted, notifications work
- **Android 13+:** POST_NOTIFICATIONS required (runtime permission)
- App handles both scenarios correctly
